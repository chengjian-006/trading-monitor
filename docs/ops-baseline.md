# 观潮·智能监控系统 运维与发布规范

> 适用范围：观潮·智能监控系统（个人 A 股交易监控，公网云服务器部署）。
> 定位：单人维护系统的轻量规矩——每条都能一个人执行、能检查。这是影响真实交易决策的生产系统，规范的底线是"生产不当测试场"。
> 版本：v1.0（2026-07-17 定稿）。既有惯例首次成文，回滚流程为本次新设计。

## 一、环境与拓扑

### 1.1 部署拓扑

| 组件 | 位置 / 规格 | 说明 |
|---|---|---|
| 应用服务器 | 华为云 ECS `124.71.75.5`，内存约 3.5G | 单机承载全部服务，登录方式 `ssh -i D:\财务管理\trading-keypair.pem root@124.71.75.5` |
| 应用目录 | `/opt/trading-monitor` | 后端源码 + `venv/` + `frontend/dist` + `site/dist` |
| 进程模型 | systemd 服务 `trading-monitor` → `venv/bin/uvicorn --workers 1`，监听 8888 | **单进程单 worker**：前台 HTTP、APScheduler 全部后台任务、aiomysql 连接池共享同一事件循环，任何同步阻塞会冻全站 |
| Web 入口 | nginx 80 端口，裸 IP `default_server` | 静态文件直出（vite hash 长缓存）、`/api/` 反代 8888、`/ws` WebSocket。域名备案中，备案通过后启用 `nginx-site.conf` 分流官网/应用子域，届时上 HTTPS |
| 数据库 | 火山引擎云 MySQL（`vedbm-*.volces.com`，库 EmailSender） | **跨云走公网**：单条 SQL 往返约 44ms，建新连接约 280ms。连接池 minsize=5 / maxsize=25 / pool_recycle=3600 |
| 大表 | `cfzy_sys_kline_cache` 601 万行（全市场日线），`cfzy_sys_kline_5m` | 前者 PK(code, trade_date) + `idx_kc_trade_date`；两表均可由回填脚本全量重建，不属"不可再生数据" |

### 1.2 配置与密钥边界（红线）

- 服务器 `/opt/trading-monitor/config.json` 是**生产专属**：DB 连接、飞书 webhook、PushPlus token、博主 cookie 等敏感凭证全在其中，已 gitignore，本地那份是空壳。
- 红线三条：**部署永不上传覆盖它**（deploy.ps1 已显式排除）；**修改前先备份 + 与维护人确认**（先例：`config.json.bak.20260621`）；**内容不进 git、不进日志、不进聊天记录**。
- 违反的真实代价：2026-06 一次部署把生产 config.json 冲空，PushPlus token 丢失导致推送静默失效，事后才发现。

### 1.3 排"系统慢"的固定顺序

先看跨云 44ms 税（N+1 / 串行多查询被乘 N 倍放大），再看单 worker 是否被同步调用阻塞，最后才怀疑代码本身。行情刷新 2.8s 超时是盘中常态基线，排查别拿午休时段对比。

## 二、发布流程与检查单

### 2.1 发布前校验闸（不过不上）

| 项 | 命令 | 通过标准 |
|---|---|---|
| 后端测试 | `pytest` | 全过 |
| 后端可导入 | import / ast 检查 | 不报错 |
| 前端类型 + 构建 | `vue-tsc && vite build`（即 `npm run build`） | 干净通过 |
| 模型/参数类改动 | 独立样本 OOS 复验 | 零重叠样本不翻车才上（过拟合教训成规） |

任一项红：停下修，不 commit、不 push、不部署。半成品不上线。

### 2.2 部署流程（deploy.ps1 六步）

1. **本机 build 前端**（`frontend/`）。前端一律本机 build——2026-07-05 事故后铁律：服务器现场 npm install + vite build 吃满 3.5G 内存导致系统卡死、SSH/后端全挂。
2. **本机 build 官网**（`site/`，独立 Vite 工程，无则跳过）。
3. **tar 打包整个工作区**，随包带上本机 build 好的 `frontend/dist`、`site/dist`。排除清单：`node_modules`、`__pycache__`、`.git`、`.claude`、`trading.db`、**`config.json`（见 1.2 红线）**、`bt_cache`（防本地旧回测缓存覆盖服务器新缓存）。
4. **scp 上传**到 `/opt/trading-monitor`。
5. **服务器解压 + 装 Python 依赖**：依赖必须装进 `venv/bin/pip`（systemd 跑的是 venv），装到系统 pip 服务看不到、新依赖哑火（2026-06-30 pywencai 中招）。服务器不再跑任何前端 build。
6. **`systemctl restart trading-monitor`**。

注意：tar 打的是**整个工作区**（非 git HEAD），未提交的 WIP 会一并上线——这是默认且正确的行为，服务器状态 = 最近一次工作区状态；只想上已提交内容时先 `git stash`。手动打包必须 `tar -C <项目根绝对路径>`，裸 `tar .` 曾因 cwd 漂移静默打错目录造成假部署。

### 2.3 部署后验证检查单（硬闸，四项全过才算"部署完成"）

| # | 检查项 | 方法 | 不过怎么办 |
|---|---|---|---|
| 1 | 服务 active | `systemctl is-active trading-monitor` | journalctl 排查，起不来即回滚 |
| 2 | **新代码真上线** | ssh grep 关键改动在服务器源码；前端核对 `dist/index.html → index-*.js → 目标 chunk` 指向新 hash | 查打包目录、查孤儿文件（见下），重新部署 |
| 3 | 关键接口可用 | 无鉴权 curl 改动涉及的接口：401 = 路由已上，404 = 没上 | 回到 #2 排查 |
| 4 | 没误删误覆盖 | config.json 未被覆盖（核对 mtime）；服务器没有把 build 搞挂的孤儿文件 | 恢复备份 / 手清孤儿文件 |

"服务 active" ≠ "新代码真上线"——本系统被假部署咬过两次（打错目录、孤儿文件致 dist 没更新），所以 #2 是四项里最不能省的。

**孤儿文件陷阱**：tar 解包只增不删，本地已删除的源文件会在服务器残留；若残留文件 import 了已不存在的模块，会把服务器侧构建搞挂。征兆：报错的文件本地 grep 不到、`git ls-files` 查不到。处置：确认后 ssh 删除该文件。

### 2.4 回滚流程（新设计，基于 git revert + 重新部署）

现状没有回滚标准，定如下轻量流程——不搞多环境、不搞镜像版本库，靠 git 历史 + 既有部署脚本兜底：

1. **判断回滚范围**：只回代码。config.json 不在回滚范围（它不随包走）；数据库不回滚（数据问题走第六章数据事故 runbook，改过表结构的先评估新表结构对旧代码是否兼容，不兼容则只能向前修复）。
2. **`git revert <坏提交>`**（一次上线含多个提交就逆序 revert 多个）。用 revert 不用 reset——保留完整历史，changelog 与事后复盘都有据可查。
3. **补一条 changelog.ts 回滚记录**（tag: 'fix'，写明回滚了什么、为什么）。
4. **走完整发布流程**：2.1 校验闸 → deploy.ps1 → 2.3 四项验证。回滚也是一次发布，闸门一个不少。
5. **紧急旁路**（服务在流血、来不及 build 全量包时）：仅前端问题可只传旧 dist 覆盖 `frontend/dist`；仅后端问题可 ssh 直接 revert 单文件 + restart。旁路只用于止血，止血后 24 小时内必须补走一遍标准流程，让服务器状态与 git 对齐。

## 三、数据库备份与恢复规范

### 3.1 备份分层

| 层 | 内容 | 频率 | 责任 |
|---|---|---|---|
| 云厂商自动备份 | 火山引擎 RDS 实例级全量 | 每日自动，保留 ≥7 天 | 确认控制台开关处于开启状态，**每季度登录核查一次**（云配置会漂移） |
| 操作前手动备份 | 破坏性操作影响的行 | 每次破坏性操作前，无例外 | 见 3.2 |
| 可再生数据 | kline_cache / kline_5m 等行情缓存 | 不备份 | 有回填脚本可全量重建，不占备份预算 |

### 3.2 破坏性操作前手动备份（既有先例成文为铁律）

任何 DELETE / UPDATE 存量业务数据、迁移表结构、清空重建，操作前必须：

1. **导出受影响行为可原样恢复的格式**（SQL INSERT 或 JSON dump），文件头写明：备份时间、备份原因、恢复方法。
2. **存入 `docs/backups/`** 并随仓库提交（含敏感个人交易数据的酌情只存本地，但必须存在）。
3. 先例即模板：`docs/backups/模拟盘数据备份-5分钟回放重建前-20260714.md`（含"如何恢复"段 + 全量 INSERT）、`docs/signalid_migration_backup_20260530.json`、脏交割单清理前的 `_backup_dirty_trades_20260607.json`。
4. 清理类脚本一律先 **dry-run 逐条核对，再 --apply**（`_fix_dirty_trades.py` 模式）。

### 3.3 恢复演练

- **每半年一次**：从云自动备份或最近一份手动备份，恢复到本地库（本地 Python 环境已理顺，可直连验证），核对行数与关键账目（如持仓净额对齐真实持仓）。
- 每次真实数据事故的恢复过程本身计为一次演练，当期免做。
- 演练不通过（备份不可恢复、格式缺列）= 高优先级修复项，比新功能优先。

## 四、监控与告警基线

### 4.1 已有监控（现状成文）

| 监控点 | 机制 | 时点 |
|---|---|---|
| 数据源健康 | data_health 三类事件（指数分时冻结回放 / 全市场快照无涨跌幅 / 10 点后日 K 缺今日 bar），即时推送，每类每日最多一次 | 盘中，flush 挂 300s 数据体检闸门 |
| 数据体检 | check_data_sanity | 每 300 秒 |
| 行情陈旧自愈 | 盘中扫 quote_updated_at > 150s 的票单独补刷 | 每 90 秒 |
| 信号质量 | signal_eod_audit：收盘真实日线复核当日全部信号，存疑标记 + 推送（不自动删） | 每交易日 17:00 |
| 系统健康汇总 | 当日累积系统故障（数据源交叉校验偏差 / 抓取中断等）合成一条盘后推送，无异常不推 | 每日 21:00 |
| 模型胜率重算 | 5 分钟真实可成交口径重算全部买入模型胜率（依赖 20:00 的 5 分钟 K 线追加完成） | 每晚 21:00 |
| 推送健康度周报 | 本周降噪动作统计 | 周五 17:10 |

### 4.2 缺口与基线要求（待补，按序做）

1. **服务存活外部探活（最高优先级）**：现有全部告警都由服务自己发出——**服务挂了就是静默**，2026-07-05 宕机即无人知晓直到人工发现。基线：站外（本机计划任务或免费拨测服务）每 5 分钟 curl 一个无鉴权接口，连续 2 次失败告警到手机。
2. **磁盘水位**：app.log 轮转 30 天 + bt_cache + tar 归档残留会累积。基线：磁盘使用率 ≥80% 告警，可并入 21:00 系统健康汇总（服务内自查磁盘不受"服务挂了"悖论影响，因为磁盘满通常先于服务挂）。
3. **证书到期**：当前裸 IP HTTP 无此项；域名备案通过、启用 HTTPS 当天，同步上到期 <14 天告警（并入健康汇总）。

## 五、日志规范

- **留存**：`app.log` 按天轮转（TimedRotatingFileHandler, midnight），保留 30 份；每日 03:30 定时任务清理 30 天前的轮转文件与 DB 操作日志（`cfzy_biz_operation_logs`）。30 天是既定基线，够覆盖一个完整的月度复盘周期，不再延长。
- **敏感信息负面清单（不入日志）**：config.json 中任何凭证（DB 密码、webhook、token、cookie）；推送通道签名密钥；交割单原始文件内容（可记行数与校验结果，不记逐行数据）。新增日志语句自查一遍此清单。
- **诊断留痕**：数据事故排查中间产物（dump、比对结果）放 scratchpad 或 `docs/backups/`，不混入应用日志。

## 六、事故响应 runbook

三类事故，每类先给判断特征，再给处置步骤。共同原则：**先止血、再备份、后清理、末复盘**；清理一律 dry-run 后 apply。

### 6.1 数据事故（脏数据入库 → 下游连锁算错）

**判断特征**：某只票成本/持仓量明显异常、逆势触发成本类卖点、报"你的持仓"但实际未持有。
**先例**：2026-06-10 交割单整行列错位（成交金额落进价格列，成本算成 11332 vs 真实 224，假浮亏止损天天误报）；2026-06-29 重复导入（同笔成交两种格式 trade_date 漂移绕过唯一键，虚增 700 股幽灵持仓触发假异动推送）。

处置步骤：
1. **定位脏数据范围**：按导入批次（imported_at）圈定行；跑 `_diag_verify_costs.py` 全持仓成本/现价偏离体检确认波及面。
2. **备份受影响行**（3.2 铁律），再 dry-run 删除脚本逐条核对 → apply。
3. **重建派生数据**：`rebuild_user_rounds` 重建交易回合；核对净额对齐真实持仓。
4. **连清衍生状态（三处检查单）**：信号表误报记录、模拟账户假持仓/流水（退回资金对账）、rally_track 等跟踪表——0612 曾漏清跟踪表，僵尸跟踪两天后又爆一条假止损推送。买入信号的衍生状态从来不止一处。
5. **护栏下沉源头**：每次数据事故收尾必须在解析/入库层加校验（先例：金额自洽校验、成本偏离 5 倍跳成本类卖点、日期无关成交指纹去重），只删数据不加护栏 = 没修完。

### 6.2 误报事故（信号/推送批量错误）

**判断特征**：多条信号触发价与真实行情对不上、竞价时段涨跌家数 0/0、指数"急跌"但当日实际上涨。
**先例**：2026-06-12 排查出 24 条误报（K 线序列错位 / 竞价数据假象 / 分时冻结回放 / 假成本），全部删净并撤销假持仓退回资金。

处置步骤：
1. **先修根因 bug 并部署**，再清数据——否则边清边产新误报。
2. **逐条核对**：最硬的判别锚点是 detail 里的触发价，反推检测器"当时认的昨高/昨收" vs 真实值；盘中口径成立的左侧信号 EOD 不复现 ≠ 误报，别错杀。
3. **dry-run 列出待删清单 → 人工过目 → apply**，删后确认残留 0。
4. **走 6.1 第 4 步的三处衍生状态检查单**（模拟持仓、跟踪表）。
5. **确认自动复核覆盖**：17:00 EOD 复核对该类误报是否有指纹可查，没有就补规则——0612 之后误报排查已从人工普查变成每日自动复核，新型误报应进同一张网。

### 6.3 服务宕机

**判断特征**：页面打不开、推送整体静默、SSH 连不上。
**先例**：2026-07-05 服务器现场 build 前端吃满 3.5G 内存，系统卡死、SSH/后端全挂，只能云控制台强制重启。根治措施（前端一律本机 build）已固化进 deploy.ps1。

处置步骤：
1. **SSH 能通**：`systemctl status trading-monitor` + `journalctl -u trading-monitor -n 100` 定位；`systemctl restart` 拉起；起不来且与最近发布相关 → 走 2.4 回滚。
2. **SSH 不通**：云控制台看 CPU/内存曲线（打满通常是某进程失控），控制台强制重启；起来后第一时间 `systemctl is-active` 确认服务随机自启正常。
3. **恢复后必查**：数据源健康推送是否恢复、宕机时段跨过的定时任务（EOD 复核 / 胜率重算 / 5 分钟 K 线追加）是否需手动补跑——K 线追加是幂等 upsert，次日自动补齐；EOD 复核跨日则手动补。
4. **复盘**：宕机根因若是资源型（内存/磁盘），对照 4.2 补对应监控项。

## 七、版本与变更管理

1. **版本号**：v1.7.x 单轨递增。参数微调/bugfix 走补丁号，新功能升次版本号，重大重构升主版本号。
2. **changelog 铁律**：每次改代码逻辑（信号策略、回测规则、业务功能、DB schema），必须在 `frontend/src/data/changelog.ts` 数组**头部**加版本记录（tag: new/improve/fix）。这是面向最终用户的版本页，缺一条 = 该改动不可追溯（曾因漏记丢过重大重构的追踪）。纯测试/调试脚本不触发。
3. **联动同步义务**（改了 A 必须动 B，防口径漂移）：买点模型或胜率口径变更 → 同步模型图鉴（models.ts + 回测 + 胜率覆盖）与模型推荐视图；飞书推送模板变更 → 同步 PC/手机端预览页。
4. **git 惯例**：改动按逻辑分组提交，push 到 origin/main；**push 前扫一遍密钥**（config 内容、token 字面量不进仓库）。
5. **变更节奏**：默认"改完 → 校验闸 → commit/push → 部署 → 四项验证"一条龙当日闭环；破坏性/不可逆库操作是唯一例外，先确认再动手。
6. **未闭环事项**进台账实时记录，闭环即销——变更管理的兜底是"没有丢在半路的线头"。
