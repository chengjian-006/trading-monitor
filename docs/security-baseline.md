# 观潮·智能监控系统 网络安全基线

> 适用范围：观潮·智能监控系统（FastAPI 后端 `backend/` + Vue3 前端 `frontend/` + 官网 `site/`），部署于公网云服务器（华为云 `124.71.75.5`），数据库为跨云火山引擎 RDS。
> 文档定位：本文是安全**基线制度**，不是一次性检查报告。每条基线给出「为什么要有这条（动因）→ 现状 → 基线要求 → 差距」，末尾汇总可执行的整改清单。
> 密级：内部。本文不含任何真实密钥明文，凡涉及凭据只标注位置。
> 版本：v1.0（2026-07-17 首次盘点，基于当前 `main` 分支代码实地调研）

---

## 一、为什么现在要立这份基线（威胁面概述）

这套系统有三个特殊背景，叠加起来决定了它不能按"个人小工具"的标准来防护：

1. **数据敏感**：系统里存的是真实交割单、持仓成本、已实现盈亏、模拟账户资金——属于个人财务隐私。一旦泄露，损失的是钱和隐私，不是几张图表。
2. **公网裸露**：服务器有公网 IP，数据库还是跨云访问（应用在华为云、库在火山引擎），大量流量要过公网。任何一段明文传输，都在公网上可被截获。
3. **即将对外开放**：目前是自用，但产品定位是未来对外。一旦多人使用，"自用系统可以接受"的取舍（长效 token、部分接口不限流）会立刻变成对外的责任。**这份基线就是为"从自用切换到对外"提前划线。**

**总体判断**：代码层的安全习惯整体在线——SQL 全参数化、端点鉴权基本全覆盖、密钥不入库、密码强哈希、历史事故都留了针对性护栏。真正的短板集中在**传输层**（数据库和 Web 都还是明文）和**几个鉴权漏配的接口**，这些是对外开放前必须先堵的。

按风险高低，当前最需要关注的四类问题：

| 风险 | 一句话说明 | 影响 |
|---|---|---|
| 生产数据库公网明文连接 | 跨云 MySQL 连接未加密，口令和全部交割单/持仓数据明文过公网 | 高 |
| 部分接口无鉴权 | 5 个行情接口 + 股票搜索 + 问财观点写入接口，匿名可访问 | 高 |
| 传输层无 HTTPS | 登录密码、JWT 令牌、上传的交割单明文过公网（卡在 ICP 备案） | 高 |
| 第三方推送明文 HTTP | PushPlus 通道用 HTTP 传输，推送内容含持仓成本价 | 高 |

## 二、认证与授权基线

**动因（内部痛点驱动）**：本系统曾把 JWT 签名密钥硬编码在源码里并随仓库进了 GitHub 公开历史（v1.7.568 已修）。密钥一旦泄露，攻击者可伪造任意用户身份——这是踩过的坑，也是本章存在的直接原因。

### 现状（做得好的）

- **JWT 密钥已运行期化**：不再硬编码，优先从 `config.json` 读取，缺失则自动生成随机密钥并回写（`backend/core/auth.py:15-40`）。当前本地密钥已是 64 位随机串，与泄露的旧值不同，即已轮换。
- **密码强哈希**：PBKDF2-HMAC-SHA256、10 万轮、每用户独立 32 字节随机盐（`backend/core/auth.py:47-56`）。合格标准做法。
- **登录防爆破**：按「IP+用户名」计数，10 分钟内失败 5 次锁 15 分钟，返回 429（`backend/routers/auth.py:15-88`），有回归测试覆盖。
- **单点登录踢人**：令牌带版本号，改密码/重登可让旧令牌立即失效（`backend/core/auth.py:81-92`）。
- **水平越权已封堵**：所有带资源 ID 的读写都按 `user_id` 过滤，抽查未发现可跨用户访问他人数据的接口。
- **两级角色**：普通用户 / 管理员，用户管理、系统配置、定时任务等敏感面均要求管理员权限。
- **无自助注册**：建用户仅管理员可为，减少了对外攻击面。

### 现状（有差距的）

- **5 个行情接口无鉴权**（疑似遗漏，非有意）：`/api/kline/batch-intraday`、`/api/kline/{code}/intraday`、`/api/kline/{code}/big-orders`、`/api/kline/{code}`、`/api/kline/{code}/week`（`backend/routers/kline.py:39,108,185,398,423`）。匿名可爬全市场 K 线库、自选池分时，也可被当免费代理刷爆外部数据源配额、加速 IP 被封。与全站"接口全需登录"的口径不一致。
- **股票搜索接口无鉴权**：`GET /api/search`（`backend/routers/search.py:8`）转发外部数据源，同样可被刷。
- **问财观点写入接口零鉴权零限流**：`POST /api/wencai/opinion`（`backend/routers/wencai.py:324`）于 2026-07-16 主动去掉了令牌校验，任何人可无限写库（单条上限 5 万字），存在刷库/存储型 DoS 风险。属当事人明知的取舍，对外开放前必须至少加 IP 限流。
- **令牌存于浏览器 localStorage**（`frontend/src/stores/auth.ts`）：XSS 可窃取。这是业界常见做法，但在无 CSP 兜底时偏弱。
- **访问令牌有效期 7 天**：偏长，靠令牌版本号机制部分补偿。自用可接受，对外应缩短并配刷新令牌。
- **管理接口降级**：`/api/admin/lark-templates` 只要求登录、未要求管理员（`backend/routers/lark_templates.py:25,33`），与 `admin` 命名不符（只读模版预览，影响小）。
- **常量时间比较缺位**：`verify_password` 及若干共享密钥比较用 `==` 而非 `hmac.compare_digest`（`backend/core/auth.py:56`、`backend/routers/blogger.py:47`、`wencai.py:147`），理论上的时序侧信道，实际极难利用。

### 基线要求

1. **默认全鉴权**：新增任何返回数据或写库的接口，默认挂 `Depends(get_current_user)`；确需公开的（登录页概览、官网表单）必须在代码注释里写明"有意公开"及理由，并配限流。
2. **公开接口必限流**：任何免登录接口都要有防刷（IP 频率 + 日上限），参照官网申请接口的三道防刷做法（`backend/routers/apply.py`）。
3. **密钥来源单一且可轮换**：签名密钥只从配置读取，禁止任何硬编码兜底默认值。生产密钥定期轮换。
4. **敏感比较用常量时间函数**：所有密码/密钥/签名比较统一 `hmac.compare_digest`。
5. **对外开放前**：缩短访问令牌有效期（建议 ≤2 小时）并引入刷新令牌；令牌改存 HttpOnly Cookie 或配齐 CSP。

## 三、推送快捷操作签名基线

**动因（内部痛点驱动）**：飞书推送卡片带"点击即静音/关模型/标记已处理"的链接，这类链接**免登录、点击即执行**。设计不当会变成"谁拿到链接谁就能替用户操作"。因此对签名和防重放有专门要求。

### 现状（做得好的）

- **签名复用 JWT 密钥但做了域分离**：`"pushpref:" + 密钥`，签名不能反过来伪造 JWT（`backend/services/push_pref.py:25-27`）。
- **六要素全入签**：用户、动作、目标、天数、过期时间全部纳入 HMAC，任一篡改即失效（`push_pref.py:50-52`）。
- **有过期时间**：链接 48 小时失效，过期时间也在签名内防篡改续命；无过期时间的旧链接一律拒绝。
- **常量时间比较**：验签用 `hmac.compare_digest`（`push_pref.py:59-60`）。
- **权限被白名单圈死**：签名只是一次性授权凭证，能触达的操作被严格限定在"推送偏好"域内，无法借道做其它事。

### 现状（有差距的，均属已知取舍）

- **48 小时窗口内可重放**：无一次性消费（nonce）机制，同一链接可反复点击。缓解在于所有动作幂等（重复静音结果相同），危害限于"攻击者拿到链接可替用户静音提醒"，属低危。
- **走 GET 方法**：签名和参数在 URL 里，会进服务器访问日志、浏览器历史（`push_pref.py:79-80`）。原因是飞书 markdown 链接只能 GET。配合 48 小时过期，泄露窗口有限。

### 基线要求

1. **免登录操作链接必须带签名 + 过期时间**，两者都纳入 HMAC，这是硬要求，已达标。
2. **签名链接能触达的操作范围必须白名单封闭**，禁止扩展到偏好域以外，尤其禁止触达资金/持仓变更类操作。
3. **对外开放前**：为高价值操作补一次性消费（nonce）机制，消除重放窗口。

## 四、密钥与配置管理基线

**动因（混合驱动）**：内部踩过硬编码密钥进公开仓库的坑；对外后，密钥管理也是审计必查项。

### 现状（做得好的）

- **生产配置不入库**：`config.json` 及其备份被 `.gitignore` 排除（`.gitignore:10-13`），全历史扫描确认从未被提交；`config.example.json` 全是占位符。
- **部署排除生产配置**：`deploy.ps1:76` 打包时明确排除 `config.json`，防止覆盖生产密钥。
- **配置接口已脱敏**：管理员读取配置时，密钥类字段统一打码，回填用哨兵机制不会覆盖真值（`backend/routers/config.py:17-68`）。
- **全历史密钥扫描基本干净**：数据库连接串、飞书/企微 webhook、PushPlus 令牌、同花顺 cookie、云厂商 AK/SK 均无真实值泄露；仅有的两个东财公开参数属网页公开嵌入值，非私有凭据。

### 现状（有差距的）

- **旧 JWT 密钥永久留在 GitHub 公开历史**：远端为公开仓库，旧硬编码密钥字符串虽已从当前代码移除，但历史提交里永久可见。**需登录生产服务器核实 `config.json` 的 `jwt_secret` 确非旧值**（本地已轮换，生产大概率已换，但值得确认一眼）。
- **生产服务器信息明文**：生产 IP + root 用户名 + SSH 私钥本地路径出现在 `deploy.ps1:9-12` 等已跟踪文件（私钥本身不在库，但 IP+root 缩小了攻击面搜索范围）。
- **本地散落明文数据库口令**（均未进 git，但在开发机磁盘上明文，有误提交/误传风险）：`_diag.py`、`backend/scripts/_watch_paper_fills.py`、`.claude/settings.local.json` 中嵌了含明文口令的诊断命令。
- **无环境变量注入能力**：所有密钥只能来自磁盘上的 `config.json`，无法走环境变量（`backend/core/config.py`），对容器化/12-factor 部署不友好，也意味着密钥安全完全依赖该文件的权限。

### 基线要求

1. **禁止任何真实密钥进源码**：提交前扫描密钥（延续现有 push 前扫描惯例）。
2. **生产密钥与仓库物理隔离**：只存生产 `config.json`，文件权限收紧到属主只读。
3. **本地诊断脚本禁止硬编码口令**：一律改为从 `config.json` 读取，消除明文副本。
4. **密钥泄露即轮换**：任何密钥一旦可能暴露，立即在生产轮换，不依赖"历史没人看"。
5. **建议**：给配置加载增加环境变量覆盖能力，降低对磁盘明文文件的依赖。

## 五、输入与数据校验基线

**动因（内部痛点驱动）**：曾发生交割单导入"整行列错位"脏数据事故，把成交金额落进价格列，导致成本算错、触发假止损告警。数据入口的校验是用真实事故换来的。

### 现状（做得好的）

- **SQL 注入面基本封死**：全项目不用拼接式 SQL，统一走参数化占位符，收口在 4 个数据库 helper（`backend/models/repo/_db.py:11-39`）。动态拼接仅出现在 IN 子句占位符、白名单列名等安全场景，参数化覆盖率接近 100%，未发现可从接口触达的注入点。
- **金额自洽护栏在岗**：交割单导入时校验"金额≈价×量"，不符拒收整行（`backend/services/trade_analyzer.py:230-234`），三条解析路径共用，正是那次事故的针对性补丁。
- **上传口全部有大小限制、内存解析不落盘、需登录**：交割单 xlsx（10MB）、同花顺自选 XML（5MB，用标准库解析无 XXE 风险）、OCR 截图（10MB）。
- **外部数据入库前清洗**：抓取的行情统一数值化 + 丢弃坏行，另有运行时数据自检和收盘复核作纵深防御。
- **请求体普遍走结构化校验**：15 个路由文件、51 处校验模型，官网公开接口甚至带字段长度约束。

### 现状（有差距的）

- **上传文件无类型校验**：交割单导入不校验扩展名/内容类型，直接尝试解析（`backend/services/trade_analyzer.py:162-171`）。后果轻（解析失败即返回未识别），但应补前置校验。
- **配置更新接口收裸字典**：`update_config` 直接合并写文件、无结构化校验（`backend/routers/config.py:72`），仅靠管理员权限缓解。
- **任意本地路径读取**：同花顺导入允许用户指定任意服务器本地路径作为查找目录（`backend/routers/ths.py:121-125`），登录用户可探测服务器目录结构。单用户风险低，对外需收敛为固定目录白名单。
- **Webhook 存在轻度 SSRF**：用户可自设 webhook 地址由服务器主动请求，无地址白名单，可指向内网（`backend/models/repo/users.py:35`）。
- **股票代码校验偏弱**：加自选只做补零、不强制 6 位数字，可入脏数据（无注入风险）。

### 基线要求

1. **所有用户输入进 SQL 必须参数化**，禁止字符串拼接值，禁止列名/表名直接取自用户输入——已达标，须保持。
2. **上传必须校验类型 + 大小 + 内存解析**，敏感财务数据导入必须有自洽性校验护栏。
3. **服务器主动发起的外网请求（webhook）必须走地址白名单**，禁止指向内网/元数据地址。
4. **不接受用户直接指定服务器文件系统路径**；确需时限定到固定白名单目录。

## 六、Web 安全基线

**动因（外部合规 + 对外开放驱动）**：HTTPS、安全响应头、限流是对外 Web 服务的通用底线，也是安全审计的必查项。

### 现状（做得好的）

- **同源架构，无跨域风险**：前后端同域部署，未启用跨域中间件，不存在"通配符 + 携带凭据"的危险组合。
- **官网接口最小暴露**：官网域名下只精确开放申请接口一条，系统接口摸不到（`nginx-site.conf:54-60`）。
- **应用端口不直接暴露**：后端绑定 `127.0.0.1`，只经 nginx 反代 `/api/` 和 `/ws`。
- **登录限流 + 官网表单三道防刷**已在岗（见第二章）。
- **接口文档端点实际不可达**：`/docs` 等未经 nginx 反代，公网访问落到前端页面。

### 现状（有差距的）

- **全站无 HTTPS**：nginx 仅监听 80 端口，HTTPS 卡在 ICP 备案（备案后 certbot）。后果：登录密码、JWT 令牌、上传的交割单当前全程明文过公网。**这是对外开放前的头号传输层短板。**
- **零安全响应头**：`X-Content-Type-Options`、`X-Frame-Options`、`CSP`、`HSTS` 一个都没有。加几行 nginx 配置即可低成本补齐（HSTS 需等 HTTPS）。
- **前端富文本渲染无统一净化**：全站 13 处直接渲染 HTML、无净化库。多数数据源可控或已自行转义，但少数信号/模版页原样透传后端/AI 生成的 HTML（`frontend/src/views/SignalView.vue`、`LarkTemplateView.vue`），构成"外部数据→AI→报告→渲染"的理论存储型 XSS 链，中低风险。
- **接口文档未显式关闭**：靠 nginx 路由屏蔽而非显式关闭，建议显式关闭做双保险，防未来配置变更或裸 IP 直连时暴露接口清单。
- **WebSocket 令牌走 URL 参数**：会进访问日志（`backend/routers/ws.py:10`），轻微。

### 基线要求

1. **对外服务必须全站 HTTPS**，HTTP 强制跳转，备案通过后立即上 TLS。
2. **必配安全响应头**：`X-Content-Type-Options: nosniff`、`X-Frame-Options: DENY`、`Content-Security-Policy`、HTTPS 后加 `HSTS`。
3. **富文本渲染必须统一净化**：引入净化库，对所有渲染 HTML 的位置统一处理，不依赖各页自觉转义。
4. **生产显式关闭接口文档端点**。
5. **令牌不走 URL 参数**，改走消息体或握手头。

## 七、日志与敏感数据基线

**动因（数据敏感驱动）**：系统存的是个人财务数据，日志和第三方推送是最容易无意间泄露它们的两个出口。

### 现状（做得好的）

- **日志不落敏感内容**：无请求体日志中间件，交割单导入只记条数不记内容，失败提示不含密钥值。
- **配置接口脱敏**已在岗（见第四章）。
- **导出面等于认证面**：xlsx 导出是纯前端生成，后端无导出端点，数据来自需登录的接口；交割单导入端点全部要求登录。

### 现状（有差距的）

- **敏感数据明文存储**：交割单、交易回合（含已实现盈亏）、模拟账户资金均为明文列，无字段级加密。自用可接受，但叠加"库在公网、连接无加密"后风险被放大。
- **第三方推送含持仓成本**：持仓守护类推送会明文带真实成本价和浮盈比例（`backend/services/holding_guard.py:182-195`），且 **PushPlus 通道走 HTTP 明文**（`backend/services/notifier.py:35`），令牌和含成本的内容全程明文过公网。改为 HTTPS 即可（官方支持）。
- **部署包裹挟本地数据**：打包不排除仓库根散落的交割单复盘 xlsx / csv / 日志，会被上传到生产服务器（不对外服务，但属不必要的数据外溢）。

### 基线要求

1. **日志禁止记录**：密码、令牌、密钥、完整配置、请求体中的财务数据——已达标，须保持。
2. **所有第三方推送通道必须走 HTTPS**，含账户成本/资金的内容尤其如此。
3. **推送内容做敏感分级**：评估持仓成本是否必须出现在第三方通道，能不带则不带。
4. **部署打包只含代码**，排除一切本地数据文件（xlsx/csv/日志）。
5. **对外开放前**：评估交割单等核心财务数据的字段级加密。

## 八、依赖与供应链基线

**动因（外部合规驱动）**：依赖漏洞是对外系统被攻破的常见入口，供应链审计是安全合规的标准项。

### 现状（做得好的）

- **用维护中的 JWT 库**（PyJWT，非已弃维护的 python-jose），HTTP 客户端统一用 httpx。
- **前端锁文件已提交**：前端与官网都有 lock 文件，安装可复现；锁定的构建链版本较新。
- **前端本机构建**：服务器不跑前端构建，减小服务器侧供应链面。
- **部署走官方源装进虚拟环境**，无第三方镜像源。

### 现状（有差距的）

- **Python 依赖全部浮动版本、无锁定**：`requirements.txt` 全部用"大于等于"下限约束，无锁文件/哈希。每次部署实际装什么由当天上游决定，版本漂移不可复现——这是最主要的供应链风险。
- **零依赖审计与 CI**：无自动化依赖审计、无持续集成流水线，校验完全靠本地手工。
- **前端 Excel 库无修复版**：`xlsx` 锁在 0.18.5，其已知漏洞的修复版只在自建渠道发布、npm 装不到。缓解在于本项目只用它导出已认证数据、不解析外部文件，暴露面小。

### 基线要求

1. **生产依赖必须锁定版本**：Python 侧引入锁文件（pip-tools/uv/poetry 任一），固定版本 + 哈希，部署照锁文件装，保证可复现。
2. **建立依赖审计惯例**：定期跑依赖漏洞扫描，接入 CI 自动化。
3. **高危无修复依赖做替换评估**：如前端 Excel 库，评估替换或严格限定其只处理可信数据。

## 十、检查机制

1. **新接口红线**：任何新增接口默认要登录鉴权；公开接口必须注释说明理由 + 配限流。评审时对照本文第二章基线逐条过。
2. **提交前扫描**：延续 push 前密钥扫描惯例，禁止任何真实密钥/口令进仓库。
3. **季度复盘**：每季度按本文各章基线做一次现状对照，更新整改清单进度；每次架构或部署方式变更后即时复查传输层（HTTPS、数据库 TLS、推送通道）。
4. **对外开放闸门**：本文高优先级项（H1–H7）全部关闭，是系统从自用切换到对外的硬性前置条件，不达标不开放。
