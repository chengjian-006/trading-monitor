# 多窗口并行开工速查

多个 Claude Code 窗口改同一个仓库时照这张卡做。核心原则：**并行地改，串行地合和发。**

业界标准答案只有一个：**git worktree**（每会话独立目录+独立分支，共享同一 `.git`，文件互不可见）。

---

## 开工前先问一句

> **这个新窗口要动的文件，跟已开的窗口撞不撞？**

- **撞** → 别并行，等前一个干完再串行做（worktree 解不了逻辑冲突，并行反而赔在解冲突上）
- **不撞** → 用 worktree 并行，往下走

---

## 场景 A：单个任务（大多数时候）

不用 worktree，一个窗口干。只守一条：

- 提交别 `git add -A`，先 `git status --short` 看一眼，**只 add 自己动过的文件**（防把别的窗口半成品带进来）

## 场景 B：要并行开多个窗口

### 1. 开隔离会话（在普通终端，不是在现有 Claude 窗口里）

```powershell
claude --worktree 任务名      # 例: claude --worktree ui-style
```

需 Claude Code ≥ v2.1.49。它自动建 `.claude/worktrees/任务名/` 并在里面起新会话，直接在新会话干活。

> 原生命令不可用时，用备用脚本：`.\worktree.ps1 new 任务名`（建在 `..\tm-任务名`），完事 `.\worktree.ps1 done 任务名` 合回。

### 2. 进去后补依赖（worktree 不共享 node_modules / 不带 config.json）

```powershell
cd frontend; npm install       # 要改前端才需要
# 要跑后端连库: 把 config.json 拷进这个 worktree(gitignore 的文件不会自动带过来)
```

### 3. 各窗口独立提交

每个 worktree 在自己分支上，`git add` / `commit` 随便用，不串味。

### 4. 收尾（串行，一个一个来，别同时）

```powershell
# 挑一个先合
git checkout main; git pull; git merge 分支名
# 第二个分支先 rebase 追平再合(冲突好解)
git checkout 第二个分支; git rebase main; git checkout main; git merge 第二个分支
git push
.\deploy.ps1                   # 从 main 部署, 预检拦 脏工作区/非main/落后origin
git worktree remove .claude/worktrees/任务名   # 合完就删
```

---

## 三个必须记住的坑

1. **发版号 / 部署 / 写生产库 —— 永远串行**，绝不两个窗口同时干。生产库（火山引擎 RDS）和部署包路径是共享的，worktree 隔离不了它们。
2. **部署前先 `git fetch` 确认不落后 origin/main**，否则从旧树部署会把别人已上线的改动冲回旧版（deploy.ps1 预检兜底，但心里有数）。
3. **worktree 攒多了要清**，`git worktree list` 定期看一眼，合完即删。

---

## 一句话

> 开新窗口前问「撞不撞文件」→ 不撞就 `claude --worktree 名字`，进去补依赖、干活、独立提交 → 收尾一个一个 rebase 合回 main → 从 main 部署 → 删 worktree。
